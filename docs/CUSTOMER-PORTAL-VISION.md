## 1. 背景・課題（なぜやるか）

ラクシーレのセールスは、お客様へ **報告すべき節目が複数回** 存在する（ヒアリング後、マッチング進行中、結果が出揃ったとき、見積提示など）。

現状の問題:
- 都度の連絡は **営業コストが大きい**
- 連絡してもタイミングが合わず **お客様に繋がらない**
- 結果として **実質的に必要な報連相が履行できていないケースが多発**

お客様の関心事は「**情報を伝えてからどうなったのか？**」であり、その時計は **ヒアリング（仕入診断）を実施した日から動き始めている**。

## 2. アプローチ（本質的な解決）

「報告という行為」を **都度の連絡から、常時公開された共同作業スペースへ置き換える**。

仕入チェッカー（ai-procurement-optimizer）の既存マイページを拡張し:

- **公開する** — ヒアリング内容 / 帳票（納品書）実データ＋データ化情報 / マッチング遂行状況 / 見積
- **ログインできる状態を作る** — ヒアリング完了時点でお客様がアクセスでき、更新があればメールで自動通知
- **返してもらう** — 見積へのFB、ヒアリング内容の補足をページ上で

→ 営業＝「報告する人」から「**一緒に進める人**」へ。お客様＝「待ちの姿勢」から「**参加の姿勢**」へ。
**ラクシーレのセールス体験のリポジショニング**である。

## 3. 前提：既存資産でほぼ届く

仕入チェッカー側に必要部品の多くが **既に存在** する（ゼロ構築ではなく拡張）:

| 構想の要素 | 既存資産 | 備考 |
|---|---|---|
| お客様認証 | NextAuth v5 / Credentials（飲食店ユーザー） | マイページ稼働中 |
| 招待→ログイン | `Invitation` モデル ＋ `/invite/[token]` | 「営業からのひとこと」付きマジックリンク |
| SFA→ポータル API | **Service API v1**（`Report.createdByProxy` 代理生成） | 逆向きAPIの足場が既にある |
| お客様データ手入力 | `Recommendation.customerCurrentPrice / customerQuantity` | お客様がデータを返す体験は実装済み |
| 通知 | Nodemailer（メール）/ LINE Bot | `Restaurant.lineUserId` 保持 |
| 帳票OCR・データ化 | `Invoice` / `Recommendation`（OCR単価） | 仕入チェッカーのコア機能 |

SFA→ポータルの **既存連携は CTA問い合わせ→`/api/webhooks/pre-lead-intake`→仮リードの一方向のみ**（`ai-procurement-optimizer/src/features/leads/vegekul-sales.ts`）。相互IDは未刻印。

## 4. 確認済みの設計判断（①〜⑦）

### ① 拡張先
**ai-procurement-optimizer**。再構築中の rakushiire-v2 は本構想と無関係。

### ② お客様の単位
**オーナー／法人単位**（ポータルの `User` が複数 `Restaurant` を束ねる）。
- SFA 側対応: `User` ↔ `company`、**`Restaurant` ↔ `store` を原子単位**とし、上位は集約で導出
- 理由: 1オーナーの複数店舗がSFA上は別会社にぶら下がる等の歪みがあるため、会社単位で硬く join せず店舗単位で繋ぐ

### (f) 同一店舗の同定キー
**原則「繋がる瞬間に相互IDを刻む」。名称・住所のあいまい突合に頼らない**（誤接続＝データ取り違え事故のため）。

- **主方式（決定論的・正とする）**:
  - **(A) CTA流入時** — 既存 `pre-lead-intake` webhook を拡張し `Restaurant.id` / `User.id` を payload に追加 → SFA が `pre_lead`→`store`/`company` 昇格時も保持
  - **(B) ポータル招待時** — `Invitation` に SFA の `lead`/`company` ID を載せ、サインアップ時に紐付け
  - 相互IDは **両側に保持**: ポータル側 `Restaurant.sfaStoreId` / `User.sfaCompanyId`（SFAデータを引く鍵）、SFA側 `store.source_restaurant_id` / `source_user_id`（更新時に誰へ通知するかの鍵）
- **補助方式（過去レコードの後付け・人の確認必須・自動マージ禁止）**:
  キー優先度 = **食べログURL（最強・物理店舗一意・既にwebhook送信済）＞ email ＞ 電話 ＞ 会社名＋住所（正規化・最弱）**

### ③ 認証方式・発火
- 「ログインできる状態を作る」= **冪等な"ポータルアクセス保証"操作**（アカウント無→招待、有→紐付け、どちらも通知）
- **発火: ヒアリング完了（SFA Hook #10 / `hearing_completed_flag`）で自動**。不適切なケースは除外制御
- 新規に必要な部品: **SFA→ポータルの inbound API**（既存 Service API v1 の延長でエンドポイント追加）
- UX: **マジックリンク優先**（非IT層オーナーの「パスワード忘れ」離脱を回避）

### ④ 公開する情報の範囲と境界（最重要・セキュリティ／関係性）
- 進捗は **内部状態を抽象化したラダー** で表示:
  `ヒアリング受付 → パートナー選定中 → 見積準備中 → ご提案 →（完了）`
  - SFA の `未確認/確認中/要確認/提案可/不可/失注` を前向きな段階へ写像
- **パートナー（卸業者）名は「提案段階」でのみ実名開示**。選定中は匿名
- **絶対に見せない**: どのパートナーが辞退/NGしたか、NG理由、内部スコア、手数料率・粗利、**他のお客様のデータ**（テナント分離。(f)の相互IDで自店舗のみに厳格スコープ）
- 補足: 既存 Report は「卸業者情報（認証済みユーザーのみ表示）」を既に出しているが、それは AIカタログ突合であり、SFA営業マッチングの内部状態とは別物

### ⑤ 双方向フィードバック
**項目に紐づくスレッド型**:

| 対象 | お客様の操作 | 返り先 |
|---|---|---|
| ヒアリング内容 | 「ここ違う／補足」コメント | portal→SFA ＋ 担当営業へ Slack |
| 見積（提案） | **承認 / 要相談 / 却下 ＋ コメント**（構造化） | 同上。営業の次アクションに直結 |
| 帳票・単価 | 既存の実価格手入力を踏襲 | 既存のまま |

見積FBを構造化するのは、お客様の反応が **営業の次アクション（再提案・クロージング）に機械的に繋がる** ようにするため。

### ⑥ 更新通知（メール）の発火点
- **メールのみ・節目イベントで即時**（全更新ではなく意味のある節目に限定）
  - ポータル開放 / ヒアリング内容公開 / 進捗1段前進 / **見積到着（最重要）**
- **通知の所有者はポータル側**に集約（送信ロジック二重化・配信事故の防止）。SFAはデータを push するのみ

## 5. アーキテクチャ

```
[SFA: vegekul-sales]                      [ポータル: ai-procurement-optimizer]
 ヒアリング/マッチング/見積 ──Service API──▶ お客様マイページ
   ・出揃い判定 / 連絡記録      (push)        ・進捗ラダー(抽象化)
   ・担当へSlack通知                          ・ヒアリング内容 / 帳票
                            ◀─feedback webhook─ ・見積受け渡し＋構造化FB
                                              ・節目メール通知(即時)
        ▲                                          │
        └────── 相互ID刻印(CTA/招待時) ─────────────┘
                (Restaurant↔store / User↔company)
```

## 6. MVP ロードマップ（各段階で価値が出る）

| MVP | 内容 | 主リポジトリ |
|---|---|---|
| **0** | 顧客連絡トラッキング = 記録＋**出揃い判定**＋軽い督促。漏れの即時応急処置＋**進捗/通知の信号エンジン** | vegekul-sales（[#1765](https://github.com/vegekul/vegekul-sales/issues/1765)） |
| **1** | ポータル read-only。**相互ID刻印（CTA webhook拡張）**＋Service APIで進捗push＋マイページに**ヒアリング/進捗ラダー/帳票**＋**節目メール** | 両リポジトリ |
| **2** | 見積の共同作業。ポータルで見積受け渡し＋**承認/要相談/却下FB**→SFA | 両リポジトリ |
| **3** | 拡張。ヒアリング補足FB・発注リマインダー連携 等 | 両リポジトリ |

**全段階の最初の土台 = (f)(A) 相互ID刻印**（CTA webhook を拡張し Restaurant.id/User.id を SFA に永続化）。ここが全ての前提。

## 7. MVP 0（#1765）の位置づけ再定義

ポータルが「常時可視化＋自動メール」を担うと、#1765 の「営業へ"顧客連絡しろ"督促」は **主役から保険に降格**。ただし **#1765 の中身はポータルの心臓**になるため捨てない:

| #1765 サブIssue | 元の役割 | ポータル文脈での意味 |
|---|---|---|
| #1766 データ基盤 | 連絡記録 | そのまま顧客連絡状態の記録 |
| **#1767 判定ロジック** | 出揃い判定 | **ポータルの進捗ラダー＋節目メールの発火源** |
| #1768 記録UI | 連絡済記録 | 営業の内部記録として継続 |
| #1769 一覧タブ | 未連絡可視化 | 内部監視として継続 |
| **#1770 督促cron** | 本人DM＋エスカレ | **軽量化（保険のみ）にスコープ縮小** |

## 8. 未決事項・次の論点

- 横断認証（rakushiire-platform #2 共通 OAuth）との整合 — お客様ログインを共通基盤に寄せるか
- 特定電子メール法（送信者情報・配信停止）対応 — お客様向け自動メールに必要
- レガシー顧客のバックフィル運用（食べログURL/email 候補突合の確認UI）
- 進捗ラダーと SFA 内部状態の写像表（詳細マッピング）
- Service API のスキーマ設計（ensure-access / push-progress / push-quotation / feedback）
